## Why

`blockr.dock` has accumulated four layers of code that have nothing to do with docking: block-presentation chrome (the "sparkle UI"), cross-cutting UI primitives (design tokens, modals, selectize widgets, button helpers), the top-level page chrome (app shell, navbar, options accordion), and modal/offcanvas scaffolding (including two Bootstrap offcanvases used as invisible DOM staging areas — a UI misuse). Today, every extension that wants the same look and feel (`blockr.ai`, `blockr.io`, `blockr.sdtm`, `blockr.dplyr`, `blockr.quarto`, `blockr.dag`) must depend on `blockr.dock` even when it never touches a dock; alternative renderers (flat grid, react-flow, single-block previews) have no clean dependency to land on.

In parallel, the modal-driven add flow (`block_modal()` / `link_modal()` / `stack_modal()`) interrupts the build-a-workflow cadence on every iteration. Earlier work on a persistent sidebar (`blockr.dock@feat/sidebar-s3-dispatch`, deployed at <https://blockr.cloud/app/empty>) was well-received but landed in the wrong package. Both problems point in the same direction: extract a `blockr.ui` package that owns the cross-cutting UI surface, and shift the modal flows to a sidebar primitive.

## What Changes

- **NEW PACKAGE** `blockr.ui` sits between `blockr.core` and `blockr.dock` in the dependency graph. Imports: `blockr.core`, `shiny`, `htmltools`, `rlang`. **No `dockViewR` dependency**, no `bslib::offcanvas` dependency.
- **NEW** design tokens (`--blockr-*` palettes, shadows, typography) and base CSS (typography classes, button overrides, form overrides, error styling, issues toggle, inline-edit, popover/tooltip overrides) ship from `blockr.ui` via a single `htmlDependency`.
- **NEW** block card surface in `blockr.ui`: `block_card()` (S3 generic dispatching on block class) + helpers, the edit-block server module (`edit_block_ui()` / `edit_block_server()` / `edit_block_validator()`), and block-display helpers (`registry_metadata()`, `blks_metadata()`, `blk_color()`).
- **NEW** generic UI helpers in `blockr.ui`: selectize widgets (`block_input_select()`, `block_registry_selectize()`, `board_select()`), button helpers (`toggle_button()`, `confirm_button()`), focus helpers (`auto_focus_script()`). `off_canvas()` and `move_dom_element()` (plus the `move-element` JS handler) stay in `blockr.dock` since they have no v0 consumer in `blockr.ui` — they may be promoted later if a second consumer appears.
- **NEW** page-chrome S3 generics in `blockr.ui` (dispatching on board): `app_shell()`, `blockr_navbar()`, `options_ui()`. Each ships with a `.default` method covering today's behaviour. `board_ui.dock_board()` becomes a thin assembly that fills the shell's slots with dock-specific content.
- **NEW** sidebar primitive in `blockr.ui`: `sidebar_ui()` plus a Shiny `InputBinding` exposing `list(open, content, pinned)`; R helpers `sidebar_open()` / `sidebar_close()` / `sidebar_toggle()` / `sidebar_set_content()`; pinned mode that reflows the page; CSS-only slide; keyboard support (Esc to close, focus trap).
- **NEW** sidebar content registry in `blockr.ui`: the `sidebar_content()` S3 generic plus trigger constructors (`add_block_trigger()`, `append_block_trigger()`, `prepend_block_trigger()`, `add_link_trigger()`, `create_stack_trigger()`, `edit_stack_trigger()`, `settings_trigger()`) and built-in methods. Extensions register custom content types by exporting their own `S3method(sidebar_content, …)`.
- **NEW** block-browser content (`block_browser_ui()` / `block_browser_server()`): cards by category, search, quick-add, detailed-add accordion, keyboard navigation. Used inside the sidebar's `add_block` content.
- **NEW** runnable demo app under `blockr.ui/inst/examples/blockr-ui-demo/` exercising the default `block_ui.board`, `app_shell()`, `blockr_navbar()`, `options_ui()`, sidebar, and block browser end-to-end *without DockView* — proving `blockr.ui` is independent of `blockr.dock`, and serving as the basis for chromote-driven integration tests via the `shiny-chromote-inspect` workflow. Launched via a small `run_demo()` helper.
- **BREAKING (in `blockr.dock`)** the modal builders `block_modal()` / `link_modal()` / `stack_modal()` and their CSS/JS scaffolding (`css_modal()`, `css_modal_advanced()`) are deleted. The two staging Bootstrap offcanvases (`blocks_offcanvas`, `exts_offcanvas`) are deleted and replaced by a plain hidden `<div id="<id>-block_pool">` injected by `blockr.dock::dock_pool_ui()`. The Bootstrap settings offcanvas is replaced by `sidebar_content.settings()`. `lifecycle::deprecate_warn()` shims in `blockr.dock` re-export the moved symbols for one deprecation cycle.

## Capabilities

### New Capabilities

- `blockr-ui-foundation`: Package boundary, design tokens (`--blockr-*`), base CSS (typography, buttons, forms, errors, issues toggle, inline-edit, popovers), the `blockr_ui_dep()` htmlDependency builder, and the cross-cutting UI helpers (selectize widgets, `toggle_button()`, `confirm_button()`, `auto_focus_script()`).
- `block-card-ui`: Block card chrome (`block_card()` S3 generic + helpers), edit-block server module (`edit_block_ui()`, `edit_block_server()`, `edit_block_validator()`, `update_blk_cond_observer()`, `create_issues_ui()`), and block-display helpers (`registry_metadata()`, `blks_metadata()`, `blk_color()`). Provides the default `block_ui.board` method.
- `app-shell`: Page-chrome S3 generics dispatching on board — `app_shell()` (full-viewport layout with navbar/sidebar/content/deps slots), `blockr_navbar()` (left/right slots + optional settings gear), `options_ui()` (board options as an accordion). Each ships with a `.default` method.
- `sidebar`: The sidebar UI primitive (`sidebar_ui()` + Shiny `InputBinding` + custom message handlers + R helpers), the sidebar content registry (`sidebar_content()` S3 generic + trigger constructors + built-in content methods), and the block-browser content (`block_browser_ui()`, `block_browser_server()`) used inside `add_block` / `append_block` / `prepend_block` flows.

### Modified Capabilities

None. This is a greenfield package; existing `blockr.ui` specs do not yet exist. Downstream impact on `blockr.dock` is tracked in **Impact** below.

## Impact

- **Affected packages.** `blockr.ui` is created from scratch. `blockr.dock` adds `blockr.ui` to its `Imports` and undergoes a five-phase migration: CSS split, R-file relocation, lifecycle shims for moved exports, sidebar adoption, and final deletion of `block_modal()` / `link_modal()` / `stack_modal()` / `css_modal*()` plus the three Bootstrap offcanvases.
- **Affected dependencies.** New deps in `blockr.ui`: `blockr.core`, `shiny`, `htmltools`, `rlang`. New deps in `blockr.dock`: `blockr.ui`, `lifecycle` (for deprecation shims). `bslib` may become removable from `blockr.dock` once the offcanvases are gone (re-evaluated at Phase 5).
- **Affected APIs.** Public `blockr.dock` exports that move (and are re-exported with `lifecycle::deprecate_warn()` for one cycle): `block_card`, `block_input_select`, `block_registry_selectize`, `board_select`, plus other exports that relocate. `off_canvas()` and `move_dom_element()` stay in `blockr.dock` (no move at v0). Public `blockr.dock` exports that disappear (listed under "Breaking changes" in `blockr.dock` `NEWS.md`): `block_modal`, `link_modal`, `stack_modal`, `css_modal`, `css_modal_advanced`.
- **Affected downstream.** Existing extensions (`blockr.ai`, `blockr.io`, `blockr.sdtm`, `blockr.dplyr`, `blockr.quarto`, `blockr.dag`, `blockr-demo-*`) continue to work unchanged thanks to the lifecycle shims. They may switch their `Imports` from `blockr.dock` to `blockr.ui` later when they do not actually need DockView.
- **Out of scope.** A future `blockr.actions` package extracting the action class and triggers from `blockr.dock`; a theming API exposing tokens via Sass / `bs_theme()`; sidebar ↔ dock-panel promotion; drag-and-drop between sidebar and canvas; pin-state persistence across reloads; a JS framework (React/Vue/Svelte) inside the sidebar; a keyboard-chord system beyond Esc / arrows / Enter.
